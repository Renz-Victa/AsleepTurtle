After working on AI initiatives inside fast-moving companies, a pattern becomes obvious: first projects fail in predictable ways. Not because the technology doesn't work — but because of decisions made before a single line of code was written.

Here are the four failure modes we see most often, and how to avoid each one.

---

## Failure mode 1: The scope is a roadmap, not a project

"We want to use AI to improve our customer experience across all touchpoints."

This is a strategy, not a project. The moment you find yourself describing an AI initiative in terms of "all," "across," or "end-to-end," you've already failed to scope it.

The first AI project should be embarrassingly narrow. One input type. One output type. One team using it. One metric you're optimising for.

Narrow projects ship. Broad ones generate presentations.

**The fix:** Describe the project in one sentence: *"We are building X to help Y do Z, and we'll know it works when W."* If you can't write that sentence, keep scoping down until you can.

---

## Failure mode 2: Success is defined after the fact

You build the thing, it produces outputs, and then everyone argues about whether those outputs are good. The technical team says yes. The business stakeholders say no. Six weeks pass. Nothing ships.

This happens when success criteria aren't defined before development starts. It's not a technical problem — it's a governance problem, and it poisons otherwise solid projects.

**The fix:** Before kickoff, get written agreement on three things:

1. The primary metric you're optimising
2. The threshold that constitutes "good enough to ship"
3. Who makes the final call if it's borderline

Write it in a document. Get sign-off. Refer back to it when the disagreements start (and they will).

---

## Failure mode 3: No feedback loop from day one

A model that can't improve over time is a depreciating asset. Every day, the world changes slightly, and a static model gets slightly worse at handling it. Without a way to capture errors, label them, and use them to improve the system, you're building technical debt from the moment you ship.

Most first projects skip this because "we'll add it later." Later never comes.

**The fix:** Design the feedback loop before you design the model. Even something simple — a thumbs up/down on outputs that gets logged to a spreadsheet — is better than nothing. The infrastructure for improvement matters as much as the initial accuracy.

---

## Failure mode 4: The use case doesn't have a user

Some AI projects get built because they're technically interesting, not because anyone asked for them. A model that routes support tickets when the support team has already developed their own system. A summarisation tool for a team that reads everything in full.

**The fix:** Before scoping, spend a day with the team that will use the output. Map their current workflow. Identify the step that takes the most time and produces the most frustration. Build for that step — not for the step that seems most amenable to AI.

---

## What a good first project looks like

The best first AI projects share a few characteristics:

- **A human is already doing the task** — which means you have examples of good and bad outputs, and a baseline to beat
- **The cost of a wrong answer is recoverable** — the model isn't making final decisions, just drafts or recommendations
- **The team affected has asked for it** — they're frustrated by the current process and would welcome a better alternative
- **It can ship in under 8 weeks** — fast enough to build momentum before organisational enthusiasm fades
